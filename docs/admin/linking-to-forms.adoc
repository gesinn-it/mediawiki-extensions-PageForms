== Two-step vs. one-step

The standard way to lead users to a form is with `#forminput`.
It renders a text input where the user types a page name; submitting navigates them to the form for creating or editing that page.
This two-step approach prevents accidental overwrites of existing pages.

When the form itself sets the page title automatically (using a `page name=` formula in the form's `{{{info}}}` tag), the first step is unnecessary.
In that case use `#formlink` instead, which takes the user directly to the form without prompting for a page name.

=== Automatic page naming

In the form's `{{{info}}}` tag, the `page name=` parameter sets the title of the page being created.
Two placeholders are available:

* `<template-name[field-name]>` — replaced with the value of the named field in the named template (cannot be used in `multiple` templates).
* `<unique number>` — replaced with the lowest integer that makes the title unique.
  Variants: `<unique number;start=1>` (start from 1 instead of blank), `<unique number;random>` (random six-digit number), `<unique number;random;4>` (four-digit random number).

Example: `page name=<Quote[Author name]> quote <unique number;start=1>` would produce titles like "Ernest Hemingway quote 1".

The `page name=` value is parsed by MediaWiki, so parser functions and magic words are allowed.
The character `#` is not allowed in page titles.

== Red links

In a template, use `#formredlink` to make a link go to a form when the target page does not exist, and behave as a normal link when it does.

If the correct form for a red link is not known in advance, offer multiple choices with numbered `alt_form` query string parameters:

[source]
----
{{#formredlink:target={{{source|}}}|alt_form[0]=Source|alt_form[1]=Magazine|alt_form[2]=Online magazine}}
----

=== Form-associated namespaces

If a namespace has a default form defined (via `#default_form` on a project page), red links into that namespace automatically point to the correct form without any extra `#formredlink` calls.

=== Global red-link redirect

To send every red link on the wiki to a form-chooser page (for wikis where most pages should be created via a form), add to `LocalSettings.php`:

[source,php]
----
$wgPageFormsLinkAllRedLinksToForms = true;
----

== Preloading data

A form can open with fields already filled in.
Preloading only applies to new pages and query forms; for editing an existing page the form always shows the page's current content.

Ways to preload data:

* Set a `default` value for a field in the form definition.
* Set `preload=page-name` in the query string to preload the entire form from a wiki page.
* Set `templateName[fieldName]=value` in the query string for specific fields.
  Separate multiple fields with `&`.
* For multiple-instance templates, include the instance number: `templateName[1][fieldName]=value`.
* Use the `PageForms::EditFormPreloadText` hook for custom preloading logic. See xref:../developer/hooks.adoc[Hooks Reference].

== Passing context values

A `query string=` argument can include arbitrary parameters beyond field values.
Combined with the Extension:UrlGetParameters extension (which provides `#urlget`), a form can read those parameters and use them to adjust autocomplete sources, conditional display, or any other dynamic behavior.

== Background edits with autoedit

`#autoedit` creates or edits pages silently in the background when the user clicks a link or button.
It supports:

* Adding or removing a value from a list field: `templateName[fieldName]+=value` / `-=value` (comma-delimited lists only).
* Editing specific instances of a multiple-instance template: `templateName[instanceNumber][fieldName]=value`.

The same editing capability is available via the MediaWiki API (`pfautoedit` action), enabling bots and external scripts to update template fields in wiki pages.
See xref:../developer/api.adoc[API Reference].

== Query forms

Query forms run inside `Special:RunQuery` and are linked with `#queryformlink`.
See xref:query-forms.adoc[Creating Query Forms] for details.
