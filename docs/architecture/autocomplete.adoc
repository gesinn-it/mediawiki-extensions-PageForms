= PageForms Autocomplete Subsystem
:toc:

== Overview

PageForms provides an *autocomplete subsystem* that allows form inputs
to suggest existing values while the user types.

Autocomplete improves usability by:

* reducing typing effort
* preventing inconsistent values
* ensuring references to existing entities

Autocomplete is *not an input type itself*. Instead, it is a shared
capability used by multiple input widgets such as:

* `combobox`
* `tokens`
* text inputs configured with suggestion sources

The autocomplete subsystem spans both *client-side JavaScript logic*
and *server-side MediaWiki API modules*.

== Architectural Position

Autocomplete sits between the form input widget and the underlying data
source.

Typical execution flow:

----
Form field
→ Input type (combobox, tokens, etc.)
→ JavaScript widget
→ MediaWiki API request
→ PFAutocompleteAPI
→ Data source (SMW / category / namespace / external service)
----

This separation keeps UI components independent from storage systems.

Layer responsibilities:

[cols="1,3",options="header"]
|===
| Layer | Responsibility

| Input widget
| Provides user interface

| JavaScript layer
| Handles user interaction and AJAX calls

| API module
| Retrieves suggestions

| Data source
| Stores actual values
|===

== Client-Side Architecture

Autocomplete interactions are primarily implemented in JavaScript.

Key components:

* `PF_ComboBoxInput.js`
* `PageForms.js`
* `ext.pf.js`

The PageForms JavaScript namespace is initialized as:

[source,javascript]
----
window.pf = window.pageforms = pf;
----

This namespace hosts widget implementations such as:

----
pf.ComboBoxInput
----

== Input Initialization

PageForms widgets are initialized using the input initialization system
described in xref:input-types.adoc[Input Type System].

Inputs register initialization logic through:

----
PageForms_registerInputInit()
----

Initialization functions are stored on the form element:

----
$("#pfForm").data("PageForms")
   ├─ initFunctions
   └─ validationFunctions
----

Initialization occurs:

* when the form page loads
* when dynamic template instances are created

== ComboBox Autocomplete Widget

The most commonly used autocomplete widget is implemented in:

----
PF_ComboBoxInput.js
----

It defines the class:

----
pf.ComboBoxInput
----

which extends the OOUI widget:

----
OO.ui.ComboBoxInputWidget
----

This widget integrates with MediaWiki's OOUI system and adds asynchronous
value retrieval.

=== Event Flow

Autocomplete suggestions are triggered by user interaction events:

* `keyup`
* `focus`
* `mouseup`

Simplified flow:

----
User types text
→ keyup event
→ setValues()
→ AJAX request to API
→ suggestions displayed in dropdown
----

== AJAX Request Construction

When remote autocomplete is enabled, the widget builds a request to the
MediaWiki API.

Example:

----
api.php?action=pfautocomplete
        &format=json
        &property=Has_author
        &substr=abc
----

Parameters depend on the configured data source.

Requests are executed via jQuery AJAX and a loading indicator is shown
during the request.

== Rendering Suggestions

The API returns suggestions as JSON objects.

Example:

[source,json]
----
[
  { "title": "Albert_Einstein" },
  { "title": "Niels_Bohr" }
]
----

If display titles are available:

[source,json]
----
{
  "title": "Albert_Einstein",
  "displaytitle": "Albert Einstein"
}
----

The client widget prefers `displaytitle` when present.

== Server-Side API

Server-side autocomplete is implemented through the MediaWiki API module:

----
PFAutocompleteAPI
----

The module is exposed through:

----
action=pfautocomplete
----

Responsibilities:

* interpret request parameters
* query the appropriate data source
* format results for the client

== API Parameters

Important parameters include:

[cols="1,3",options="header"]
|===
| Parameter | Description

| `substr`
| substring typed by the user

| `property`
| Semantic MediaWiki property

| `category`
| category whose pages should be suggested

| `concept`
| SMW concept

| `namespace`
| namespace of pages

| `wikidata`
| Wikidata search query

| `external_url`
| external service endpoint

| `baseprop`
| base field for dependent autocomplete

| `basevalue`
| selected value of base field
|===

== Supported Data Sources

=== Semantic MediaWiki Properties

Example:

----
action=pfautocomplete
&property=Has_author
&substr=al
----

The API queries the SMW store for values whose textual representation
matches the typed substring.

=== Categories

Example:

----
action=pfautocomplete
&category=Authors
----

Returns page titles within the specified category.

=== Concepts

Example:

----
action=pfautocomplete
&concept=Important_authors
----

Concept-based autocomplete retrieves results from semantic queries.

=== Namespaces

Example:

----
action=pfautocomplete
&namespace=0
----

Returns pages from a given namespace.

=== Wikidata

Example:

----
action=pfautocomplete
&wikidata=Q5
----

Allows suggesting external entities from Wikidata.

== Dependent Autocompletion

Autocomplete fields may depend on another field.

Example relationship:

----
Country → City
----

Example API request:

----
action=pfautocomplete
&property=City
&baseprop=Country
&basevalue=Germany
----

The API filters suggestions based on the selected base value.

== Display Title Handling

Autocomplete results may include:

* `title`
* `title` + `displaytitle`

Example:

[source,json]
----
{
  "title": "Albert_Einstein",
  "displaytitle": "Albert Einstein"
}
----

If `displaytitle` exists, the client displays it while internally
storing the page title.

== Label Disambiguation

When results come from *categories, concepts or namespaces*, PageForms may apply
a label disambiguation step.

This behavior is controlled by:

----
$wgPageFormsUseDisplayTitle
----

The API maps page titles to display titles and ensures that labels
remain unique.

Example situation:

Pages:

----
Paris
Paris_(Texas)
----

Both may have display title:

----
Paris
----

The disambiguation logic ensures the user sees distinct suggestions:

----
Paris
Paris (Texas)
----

Internally the stored values remain:

----
Paris
Paris_(Texas)
----

This preserves readability while keeping references accurate.
